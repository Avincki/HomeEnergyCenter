Energy Orchestrator

A self-hosted Python service that minimizes electricity cost on a Belgian dynamic-tariff contract by reading every relevant home energy asset and deciding minute-by-minute whether the SolarEdge inverter should produce or be curtailed. Runs on a Raspberry Pi or any always-on PC; serves a live dashboard, a debug board, an API, and a built-in config editor.

Python 3.12 FastAPI + SQLAlchemy 2.0 SQLite + Alembic 253 tests passing Home-LAN, no cloud Belgian dynamic tariff

1. What it does

The orchestrator is the brain of a home-LAN energy setup that combines a sonnenBatterie, a SolarEdge PV inverter, several HomeWizard kWh meters, and a Belgian dynamic-tariff electricity contract. Every few seconds it polls all devices, fetches the day-ahead market price, and once a minute decides whether SolarEdge should keep producing or be curtailed via Modbus.

The economic intuition: when the wholesale injection price is negative, exporting PV power costs the household money. When it’s positive (or the battery is low, or the EV is charging), local production is profitable. A four-rule engine captures that trade-off and the inverter is set to 0 % or 100 % active-power-limit accordingly. Manual override is one click away on the debug tab.

Why self-hosted: the inverter live on the home LAN, talks Modbus on a non-routable port, and we want it to keep working when the internet drops. All assets, including the JS chart library, are vendored locally. No third-party cloud is in the loop except ENTSO-E for prices and Forecast.Solar for the PV-output forecast, both with graceful fallbacks.

2. Architecture at a glance

+------------------+ +------------------+ +-------------------+ | sonnenBatterie | | HomeWizard x4 | | SolarEdge | | HTTP /api/v2 | | HTTP /api/v1 | | Modbus TCP :1502 | +--------+---------+ +--------+---------+ +---------+---------+ | | | v v v +-----------------------------------------------------------------------+ | Tick loop (orchestrator.py) | | every poll_interval_s : read all devices -> write Reading row | | every decision_interval_s : run rule engine -> write Decision row | | -> actuate SolarEdge | +-----------------+----------------------+-----------------+------------+ | | | +-------v-------+ +-------v------+ +------v-------+ | SQLite (WAL) | | PriceCache | | SolarCache | | readings, | | ENTSO-E | | Forecast.Sol | | decisions, | | (today + 2d) | | (30 min TTL) | | source_status | +--------------+ +--------------+ +---------------+ | v +-----------------------------------------------------------------------+ | FastAPI app : / /debug /config /logs /docs + /api/* JSON | | live tiles, in-place dashboard refresh every 5 s | +-----------------------------------------------------------------------+

A single Python process. The tick loop runs as a background asyncio.Task started by FastAPI’s lifespan. The web layer reads the same SQLite file and the same in-memory caches that the loop writes to. There is no message bus, no queue, no microservices — one process, one DB file.

3. Data sources

SourceProtocolWhat we readRequired?
sonnenBatterie HTTP /api/v2/status (token) or /api/v1/status SoC %, battery power, house consumption, production, grid feed-in Yes — SoC drives Rule 1
HomeWizard car-charger meter HTTP /api/v1/data active_power_w — threshold detects "EV is charging" Yes — drives Rule 2
HomeWizard P1 (smart meter) HTTP /api/v1/data Grid import/export, informational Yes — tile + future rules
HomeWizard small-solar meter HTTP /api/v1/data Small (non-SolarEdge) PV output — proxy for irradiance, used by Rule 4 forecast Yes
HomeWizard large-solar meter HTTP /api/v1/data Second PV string — the one SolarEdge controls Optional
SolarEdge inverter Modbus TCP :1502, register 0xF001 Active-power-limit (read & write); read-back-verified on every write Yes
ENTSO-E HTTPS day-ahead REST Hourly day-ahead injection & consumption prices for BE bidding zone Yes — drives Rules 3 and 4
Forecast.Solar HTTPS public API Hour-by-hour PV-output forecast for each plane (E/W); summed total kWh today/tomorrow Optional
HomeWizard local API gotcha: the local HTTP API is off by default on every HomeWizard device. Open the HomeWizard Energy app → pick the meter → Settings → toggle Local API on. Without that, port 80 will refuse the TCP connection even though the device is reachable on the LAN.
SolarEdge inverter prerequisites (one-time, in SetApp): Modbus TCP enabled, and Advanced Power Control enabled AND committed. Without "committed", writes to 0xF001 succeed but the inverter ignores them.

4. Decision logic

The rule engine evaluates four rules in priority order. The first rule whose condition holds determines the action. Manual override sits on top — when the user picks Force ON / Force OFF from the debug tab, the auto computation is still recorded for audit, but the override decides what the inverter actually does.

RuleConditionActionWhy
Rule 1 SoC < battery_low_soc_pct (default 60 %) SolarEdge ON Recharge the battery first — price doesn’t matter when reserves are low.
Rule 2 Car charging (active_power_w > charging_threshold_w) SolarEdge ON Cover instantaneous EV demand from PV instead of importing.
Rule 3 Current-hour injection price > 0 SolarEdge ON Any positive export rate makes production profitable.
Rule 4 Forecast end-SoC during the upcoming negative-price window < battery_full_soc_pct (default 80 %) SolarEdge ON There’s headroom in the battery; produce locally and store rather than export at a loss. Otherwise OFF.

A hysteresis_pct (default 5 %) is applied around the low-SoC threshold to stop the inverter from flapping at the boundary. The previous overall state is what hysteresis is sticky against — once the rule engine has flipped to ON because the battery was low, it stays ON until SoC reaches low + hysteresis.

Forecast horizon (Rule 4): looks ahead forecast_horizon_h hours (default 12) at the cached ENTSO-E prices, finds the contiguous run of negative-injection-price hours from now, and projects how much the small (non-SolarEdge) PV string would charge the battery over that window if SolarEdge stayed OFF. If end-SoC would still be under 80 %, the engine concludes there’s room to absorb SolarEdge output too.

5. Polling & decision cadence

The orchestrator separates two cadences so the dashboard can feel live without hammering the inverter:

poll_interval_s

Default: 5 s

How often the tick loop reads every device and writes a Reading row. Drives dashboard freshness and per-source health timestamps.

decision_interval_s

Default: 60 s

How often the rule engine runs and SolarEdge gets actuated when the decision flips. Runs once on startup so we don’t sit idle for a minute, then on the elapsed-time gate.

Browser refresh

5 s, JS-driven

The dashboard polls /api/state + /api/history + /api/prices + /api/solar every 5 seconds and updates tiles + chart datasets in place via chart.update("none") — no full page reload. Pauses on visibilitychange when the tab is hidden.

If you don’t configure decision_interval_s it defaults to 60 s. The decision engine ran on every tick before this split landed; if you upgrade an existing install you may want to set both explicitly.

6. Dashboard tour

State card

The big banner at the top tells you what the inverter is doing right now. SolarEdge: ON or OFF, plus the rule that fired (battery_low_soc, positive_injection_price, …) and a one-line reason. If a manual override is active, an override: force_on / force_off chip appears next to the state.

Tiles

The KPI tiles all use the convention "positive number = good news for the household":

  • Battery SoC — %, from sonnen.
  • Battery power — positive = charging, negative = discharging.
  • House consumption — instantaneous draw, watts.
  • Car charger — HomeWizard meter active power.
  • Solar production (small + large) — wide tile with the formula small + large = total; positive = generating.
  • Grid import — positive = drawing from grid, negative = exporting.
  • Injection price — current hour’s €/kWh (negative shown red elsewhere on the chart).
  • Override — AUTO / FORCE_ON / FORCE_OFF, with an "until HH:MM" hint when set with a duration.
  • Expected solar today — Forecast.Solar total for today, kWh.

Chart

The combined chart on a single canvas, with three Y-axes:

  • Gray bars = day-ahead injection €/kWh (left axis). Red bars = negative hours. The current hour gets a cyan accent outline.
  • Green line with dark halo = battery SoC % (right axis). The halo trick keeps the line legible where it crosses bars or fills.
  • Yellow line with dark halo = measured total solar (small + large), kW.
  • Filled orange = forecast solar for the day, summed across planes (kW).

X-axis is locked to local 00:00–24:00 today. The price cache always fetches yesterday + today + tomorrow (UTC) so the chart stays fully populated across the local-day boundary.

Recent decisions

Below the chart, a table of the last 24 hours of decisions: time, ON/OFF badge, rule that fired, reason, and override mode if any. Useful to spot chatter or to audit why the inverter flipped at 03:17.

7. Debug tab

Source health

For every source the orchestrator polls, the table shows:

  • Status — OK / DEGRADED / ERROR / UNKNOWN. Whichever event (success or error) is more recent decides the badge, so a successful poll right after a failure clears ERROR immediately.
  • Last success / Last error — in local time.
  • Error message — the last verbatim error string, for diagnosis.

Below the table, a Clear all errors button POSTs to /api/source-status/clear-errors — nulls last_error_at and last_error_message for every row. Pure ack-and-clear; the next failed poll repopulates the columns.

Manual override

Pick a mode (auto / force_on / force_off), optionally set a minutes duration, hit Apply. The override expires automatically when the timer elapses; auto clears any active override immediately. The auto-decision is still computed and logged every tick while the override is active — the audit trail is honest.

Recent decisions & active config

Bottom panels: a wider 50-entry decision log, plus the active configuration rendered as JSON (with secrets redacted to ***).

8. Config & logs

/config — in-browser editor

Edit config.yaml from the browser without SSHing in. Three-column layout grouped by Devices / Decision / System. Pydantic validation errors render inline next to each input. Saves write the new YAML atomically and keep a single .bak rotation. A successful save shows a banner reminding the user that the orchestrator must be restarted for changes to take effect (the price cache, decision interval, etc. are read once at startup).

/logs — live log viewer

Streams logs/energy_orchestrator.log via Server-Sent Events. Level dropdown (debug+ … critical), substring filter, follow-toggle, pause/resume, clear; auto-trims to 1000 rows in the browser. Reconnects automatically when the rotating handler swaps the file out at midnight.

9. JSON API

All under /api. Browse /docs for the full Swagger UI.

MethodPathReturns
GET/api/stateLatest reading + decision + override + sources.
GET/api/history?h=24Readings + decisions for the last N hours.
GET/api/sourcesLast success/error per source.
GET/api/healthConfig + per-source health snapshot.
GET/api/pricesCached day-ahead prices: yesterday + today + tomorrow (UTC).
GET/api/solarCached Forecast.Solar series + per-plane breakdown + today/tomorrow Wh totals.
GET/api/logs/streamServer-Sent Event stream of the rotating log file.
POST/api/override{ mode, minutes? } — apply or clear override.
POST/api/source-status/clear-errorsNull out last_error_* on every source row.

All timestamps are emitted as UTC-aware ISO-8601 strings ("+00:00" suffix). The dashboard JS converts to local time client-side via new Date(...), so users in different timezones see local hours even if the server is in UTC.

10. Install & run

# clone and install in editable mode
git clone https://github.com/Avincki/HomeEnergyCenter.git
cd HomeEnergyCenter
python -m venv .venv
.venv/Scripts/activate         # PowerShell: .venv\Scripts\Activate.ps1
pip install -e .

# first-time DB setup
alembic upgrade head           # or: stamp 0001_initial then upgrade head if
                               # the DB was created via init_schema

# copy the example config and fill in your hosts/tokens
cp config.example.yaml config.yaml
$EDITOR config.yaml

# run
python main.py                 # equivalent to: uvicorn ... --host 0.0.0.0 --port 8000

Dashboard at http://localhost:8000/. Debug at /debug. Config editor at /config. Logs at /logs. Swagger UI at /docs.

Dry-run mode is on by default. Set decision.dry_run: true in config.yaml to suppress all SolarEdge writes while you observe what the engine would do. Keep it on for the first few days; flip to false only after you’ve confirmed the decisions are sane.

12. Raspberry Pi kiosk deployment

Three escalating levels of polish for displaying the dashboard on a wall-mounted screen:

Quick: chromeless app window no code, any OS

# Windows / Linux / macOS — strips browser chrome around the URL
chrome --app=http://localhost:8000
msedge --app=http://localhost:8000
chromium --app=http://localhost:8000

Pi kiosk mode fullscreen, no UI, autostart at boot

# Raspberry Pi OS (Bookworm desktop, LXDE-pi)
mkdir -p ~/.config/lxsession/LXDE-pi
cat > ~/.config/lxsession/LXDE-pi/autostart <<'EOF'
@xset s off
@xset -dpms
@xset s noblank
@chromium-browser --kiosk --noerrdialogs --no-first-run \
                  --check-for-update-interval=31536000 \
                  http://localhost:8000
EOF
sudo reboot

After reboot Chromium opens fullscreen with no chrome. To exit, SSH in and pkill chromium, or pop a terminal with Ctrl+Alt+T. The xset lines stop the screen blanking. For a Pi-Lite + cage setup the equivalent is a systemd service running cage chromium-browser -- --kiosk URL.

PWA polish optional

Adding a manifest.webmanifest with display: standalone plus a <link rel="manifest"> in base.html lets any modern browser offer "Install Energy Orchestrator" — once installed it launches in its own chromeless window from the Start menu / Pi launcher. Same effect as --app=URL but the user does it once via the browser’s UI instead of typing a long flag.

13. Troubleshooting

"no such column: readings.large_solar_w"

The DB schema is older than the ORM model. Either the database was created via init_schema (no alembic version stamp), or you upgraded the code and forgot to run migrations. Stop the orchestrator, then: 

alembic stamp 0001_initial    # only if alembic_version table is empty
alembic upgrade head

HomeWizard device "connection refused"

Ping works, port 80 refuses. The local API toggle is off. Open the HomeWizard Energy app → pick the device → Settings → enable Local API. Restart not required.

SolarEdge actuation "unexpected keyword 'slave'"

pymodbus 3.7+ renamed the kwarg from slave= to device_id=; by 3.11+ the deprecated alias is gone. The current codebase already uses device_id=; if you see this, you’re on an old commit or an old pymodbus.

Dashboard tile values stuck for 30+ seconds

The browser is polling correctly but no new Reading rows are being written. Check poll_interval_s in config.yaml — if it’s 30, that’s your refresh rate. Set it to 5 for live tiles. Restart the orchestrator after editing.

Dashboard chart starts at 02:00 instead of 00:00

The chart x-axis is local-day; in zones east of UTC the first hours of local-today live in yesterday’s UTC date. The orchestrator now fetches yesterday + today + tomorrow (3 days UTC) so the chart stays full. If you still see a gap, force a price refresh: curl http://localhost:8000/api/sources and check the prices source last_success_at.

Status badge shows ERROR with a stale message

Pre-fix behavior: an error within 5 minutes forced ERROR even after a successful recovery poll. Current behavior: most-recent event wins, so any successful tick after the error clears the badge. If you still want to wipe the historical "Last error" timestamp, click Clear all errors on the Debug tab.

The whole project is a single Python package with seven sub-packages (config, data, devices, decision, prices, solar, web). The tick loop in orchestrator.py is the only background task; FastAPI’s lifespan starts it on app startup and cancels it on shutdown.
